notifier.service – Simplified Service Creation¶

The tools in this module allow a programmer to spin up a service that can respond to RPC calls very easily.

First a Worker subclass needs to be defined, which has methods decorated with command. This class (the class itself) is assigned to the worker_class attribute of a Server subclass, which also needs a subscription class attribute, a string of the service name it is offering. The Server subclass gets instantiated with a count of worker instances it should create and control, a Notifier it uses to register for and receive RPC calls, and a bounds dictionary, which contains mask and value keys with integer values (more on those in a minute).

When an RPC request is made, it is always made with a object, an id, and a cmd. For the request to be routed to a particular Server, that server’s subscription must match the object, it must be true that (<the server's "mask"> & id) == <the server's "value">, and finally that the cmd in the rpc call must be "execute".

If the rpc call was made with notifier.access.execute() or notifier.access.random(), the cmd argument will be placed in the command key in the rpc request’s arguments, which will be a dictionary. The server’s Worker subclass must have that value as the name of a command-decorated method, and that method will then be used to handle the RPC and generate the response.

The bounds given to Server.__init__ control which ids get through to a server by masking with bounds['mask'] and comparing the result to bounds['value']. For a server that is the only provider of a service and command, use {'mask': 0, 'value': 0}.

class notifier.service.Worker¶

A coroutine thread that handles RPC requests.

The primary use of this class is simply to subclass it adding handler methods, and assign that subclass to a Server subclass which will actually create the worker instances and call their methods.

In a Worker subclass, any methods wrapped with notifier.decorators.command() will be used to handle RPC requests.

class notifier.service.Server(worker_count, notifier, bounds=None, loglevel=DEBUG)¶

Server, like Worker, is itself primarily useful as a base class. Subclasses of Server should get a few class-level attributes, and then be instantiated to start running in a gogreen coroutine environment.

Parameters:
  • worker_count (int) – The number of worker_class instances to run, each pulling from the work queue.
  • notifier – A Notifier (a half-notifier Publisher is not enough)
worker_class¶

The Worker subclass that will be used by instances of this Server subclass.

subscription¶

The service name (string) that this Server subclass is providing.
